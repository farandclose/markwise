The review loop between you and your AI agent is broken.

Agents hand you long, clean markdown - PRDs, specs, launch plans. Then your feedback has to make it back to the agent, and that handoff is where everything falls apart.

Markwise is the human-agent review layer for markdown.

01 - The problem

The two things you can do today

There are only two moves available, and each loses something the agent needs.

1. Reply in chat

You retype the feedback into the terminal: "in section 3, the timeline should be H2 not Q4..."

It loses the exact location and the intent. You describe a spot in prose instead of pointing at it. The longer the doc, the worse it gets.

2. Edit the markdown directly

You open the file and change the text yourself, or scribble inline notes.

The agent gets no structured signal: what changed, what is still open, what needs a response. A raw diff is not a review.

In Google Docs you comment and suggest edits right on the text, with threads and states. That fluency simply does not exist for the markdown an agent just wrote you.

02 - The reframe

This is not a markdown-comments problem

The core insight

The problem is that markdown needs comments.

Agent-written documents need a durable feedback loop that both humans and agents can understand.

Markwise is not annotation syntax. It is a workflow contract: the human leaves anchored, structured feedback; the agent revises and answers every item; the human decides what is resolved; and the document carries the whole history without polluting a normal read.

03 - The loop, in twenty seconds

Watch one piece of feedback close

launch-plan.md

##Rollout

We will launch the partner beta in Q4 2026in H2 2026, with mobile following in the new year. Pricing stays flat through the beta period.

you

We agreed H2, not Q4 - the partner beta cannot slip past summer.

agent

Changed to H2 2026. The mobile follow-up date assumed the Q4 start, so I flagged it below.

✓ resolved

The whole exchange - comment, revision, reply, resolution - is saved inside the markdown file, invisible in any normal preview.

04 - The contract

The workflow contract

  1. You leave anchored feedback - comments and suggested edits, right on the text.
  2. The agent revises the document against every note.
  3. The agent answers every item - what changed, what it pushed back on.
  4. You decide what is resolved. Closure is the human's call.
  5. The file carries the whole history - invisible in any normal markdown preview.

In-file truth

The document and all its review state are one self-contained artifact. No sidecar files, no database, no platform. Portable, version-controllable, model-agnostic.

Clean preview

All review data hides in HTML comments. GitHub, VS Code, and every normal markdown preview render the document clean.

05 - The toolkit

One CLI, one previewer

markwise preview

Review in the browser: comment, reply, suggest edits, resolve. Three themes.

markwise prompt

Bundles the doc and every open note into an instruction block any model can act on.

markwise status

Whose turn is it? Open vs resolved, waiting on you vs waiting on the agent.

markwise lint

24 rules guard the review data; --fix repairs mechanical drift, never prose.

markwise export

A clean copy with every trace of review stripped. The original is never touched.

06 - The alternatives

Why not just...

Google Docs / Notion

Your doc leaves the repo and the agent loses it. Truth moves out of the file.

PR review

Feedback anchors to diffs and lives in the forge, not the artifact. Prose is not code.

CriticMarkup

Editorial syntax, not a review loop - and raw markup leaks into every preview.

Chat

"In section 3, the timeline..." - describing locations instead of pointing at them.
